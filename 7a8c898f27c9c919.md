# LinkForge

LinkForge 是一个面向技术开发者与内容创作者的轻量级外链资源聚合与导航工具。该项目定位于解决个人或团队在多个信息源之间反复切换、查找技术文档、影视素材、设计资源或社区入口时效率低下的问题，通过提供一个可自部署的集中式链接管理面板，将分散的高质量外链进行结构化组织与快速检索。LinkForge 不存储任何实体文件或用户数据，仅作为公开可用网络资源的索引层，适用于需要构建个人书签系统、团队共享资源看板或开源项目附属导航页的场景。

## 功能概览

- **分类标签管理**：支持为每条外链分配多个自定义标签，并基于标签进行快速筛选与分组展示，便于维护大规模链接库。
- **多维度检索** 提供基于标题、描述、标签及域名关键字的全文搜索，并支持按添加时间或点击热度排序。
- **导入与导出** 支持批量导入 URL 列表（支持 CSV 与 JSON 格式），并可一键导出为 HTML 书签文件或 Markdown 表格，方便迁移与备份。
- **访问统计看板** 内置轻量计数模块，记录每个链接被点击的次数与最后访问时间，帮助识别高频资源。
- **公开与私有模式** 支持设置链接库为公开可见或需要登录 Token 访问，满足内部团队使用与个人私密收藏两种场景。
- **响应式栅格布局** 前端采用自适应卡片布局，在桌面、平板与移动设备上均能获得良好的浏览与操作体验。
- **定时健康检查** 后台内置周期性探测任务，可检测链接是否可达并标注异常状态，减少死链干扰。
- **RSS 订阅输出** 为链接库生成标准 RSS 2.0 订阅源，允许用户通过阅读器跟踪新增资源。

## 应用场景

- **个人技术书签中心** 开发者可将日常查阅的官方文档、API 参考、博客教程及 GitHub 仓库统一收录，通过标签区分技术栈（如 React、Rust、Kubernetes），替代浏览器自带书签的扁平化管理。
- **团队项目资源看板** 小型开发团队或开源项目维护者可使用 LinkForge 搭建项目相关的设计稿链接、原型工具、测试环境入口及会议纪要文档索引，减少在聊天软件中反复查找历史消息的频次。
- **内容创作者素材库** 视频剪辑师、文案写手或自媒体运营者可分类保存免版权图片网站、背景音乐库、字体下载页及竞争对手案例链接，并通过备注字段记录使用权限与到期时间。
- **教育培训辅助目录** 讲师或培训组织者可为本课程创建独立链接页，将课件幻灯片、在线练习平台、视频教程地址及参考论文集中呈现，学员无需记忆多个网址。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 18+。

```bash
# 克隆仓库至本地
git clone https://github.com/your-org/linkforge.git
cd linkforge

# 安装项目依赖（使用 npm）
npm install

# 以开发模式启动本地服务（默认监听 3000 端口）
npm run dev
```

启动成功后，浏览器访问 http://localhost:3000 即可进入初始化配置页面，按引导完成管理员账户创建与基础分类设置。

| 可选：Nginx | 1.20+ | 生产环境下推荐作为反向代理服务器，用于静态资源缓存与 HTTPS 终止 |

## 项目结构

```
linkforge/

│   ├── src/

│   │   ├── models/             # 数据模型层，封装 SQLite 表操作与查询构造器
│   │   ├── services/           # 业务逻辑层，包含健康检查调度、RSS 生成、导入导出解析

│   └── package.json            # 后端独立依赖声明（若与前端分离）

│   ├── src/

│   │   ├── views/              # 页面级组件（首页、详情页、管理后台、设置页）
│   │   ├── stores/             # 状态管理（链接列表、用户偏好、UI 主题切换状态）
│   │   ├── api/                # 与后端交互的请求封装（axios 实例、拦截器）

├── tests/                      # 单元测试与集成测试（使用 Jest / Vitest）
├── .env.example                # 环境变量配置模板（含数据库路径、端口、密钥示例）
```

## 贡献指南

欢迎并感谢任何形式的贡献，包括但不限于报告问题、提交代码、完善文档或提供翻译。请遵循以下步骤：

1. 查阅 issue 列表，确认当前未被认领的任务或提议新功能。对于较大改动，建议先创建一个讨论型 issue 以对齐设计方向，避免无效开发。
2. 从 main 分支派生新特性分支，命名格式为 `feature/简短描述` 或 `fix/问题编号`，并确保本地通过所有现有测试用例。
3. 提交代码时遵循 Conventional Commits 规范（如 `feat: 添加标签批量删除功能` 或 `fix: 修复搜索时特殊字符转义问题`），并附上清晰的变更说明。
4. 若涉及前端界面变动，请附上前后对比截图或录屏；若涉及 API 变更，请同步更新 `docs/api-reference.md` 中的请求与响应示例。
5. 发起 Pull Request 至 main 分支，等待维护者 Review。在 CI 流水线全部通过且至少一名维护者批准后，将被合并。

## 常见问题

**问：首次启动后示例数据为空，如何快速填充演示链接？**

答：项目默认不包含任何强制示例数据，以保护用户隐私。但提供了种子脚本，您可执行 `npm run seed` 从 `backend/seed/demo-links.json` 中导入一组预置的分类与链接（包含本页面资源列表中的全部 URL）。导入后可自行编辑或删除。

**问：健康检查标记链接为异常后，是否会重复通知？**

答：默认行为为每天仅记录一次异常状态并在日志中输出警告，不会发送外部通知邮件或消息。您可在配置文件中启用 `ALERT_WEBHOOK` 选项，将其指向自定义的 Discord/Slack 机器人地址，届时每次状态变更（从正常变为异常，或从异常恢复）均会触发 JSON 格式的 POST 请求。

**问：是否支持多用户权限管理？**

答：当前版本仅提供单管理员模式，所有具备访问 Token 的用户拥有相同的读写权限。若需区分只读与编辑角色，请关注后续版本的路由级权限控制功能。
