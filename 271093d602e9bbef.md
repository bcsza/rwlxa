# PaperLink Hub

PaperLink Hub 是一个面向学术研究者、技术文档工程师以及信息聚合爱好者的开源外链与资源导航工具。该项目旨在解决研究过程中文献资料分散、优质技术博客难以追溯、以及团队内部知识库链接管理混乱的问题。通过结构化的目录分类与轻量级的本地检索机制，用户能够快速建立个人或团队的高频访问资源看板。

项目本身不存储任何实际内容，仅作为指向外部优质资源的“指针系统”。其核心定位是成为技术团队内部的默认起始页（Startpage），或是个人研究者用于替代浏览器杂乱书签栏的轻量化自托管方案。PaperLink Hub 强调链接的持久性与分类逻辑的透明性，鼓励社区贡献者共同维护一个高价值、低重复的互联网资源索引。

## 功能概览

- **分类索引看板**：资源按项目批次与领域自动归类，每个外链均附带原始采集批次标记，便于追溯来源与更新周期。

- **静态链接直出**：所有收录的 URL 严格保持用户提交时的原始格式（协议、域名、路径均不做任何改写），确保链接的有效性与访问合规性。

- **项目结构即文档**：通过 ASCII 目录树直接展示文件组织逻辑，新贡献者可迅速理解资源添加规则，无需额外阅读配置文档。

- **轻量级依赖**：仅依赖标准 Python 3 环境与内置 HTTP 服务器模块，无需数据库或复杂前端框架，开箱即用。

- **协作友好型元数据**：每个外链条目支持附加备注字段（如“备用镜像”“需代理访问”），便于团队内部共享访问经验。

- **离线可浏览**：生成静态 HTML 后，所有资源列表与分类信息均可在本地文件系统中直接打开，适合内网隔离环境部署。

## 应用场景

- **学术文献速查**：研究小组可将 arXiv、PubMed、Google Scholar 等高频入口与对应镜像站统一收录至 PaperLink Hub，作为组内共享的论文检索起始页，避免每次重复输入复杂 URL。

- **技术团队知识库导航**：开发团队可将内部 Wiki、CI/CD 面板、日志平台、代码仓库地址按项目环境（开发/测试/生产）分类整理，通过本地部署的 Hub 页面统一入口，减少上下文切换时间。

- **个人书签替代方案**：针对浏览器书签栏杂乱且跨设备同步不便的问题，用户可将 Hub 部署在个人 VPS 或 GitPages 上，作为云端书签管理器，所有链接以纯文本形式维护，便于版本控制。

- **开源社区资源汇总**：开源项目维护者可将相关生态工具（如依赖镜像站、文档镜像、社区论坛）整理成 Hub 子目录，随项目仓库一同分发，降低新贡献者的环境搭建门槛。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境。请确保系统已安装 Python 3.6 及以上版本。

```bash
# 克隆仓库至本地
git clone https://github.com/paperlink-hub/paperlink-hub.git
cd paperlink-hub

# 安装依赖（项目本身无第三方依赖，仅创建虚拟环境用于隔离）
python3 -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate

# 运行内置静态服务器，默认端口 8080
python3 -m http.server 8080 --directory ./public
```

启动后，在浏览器中访问 `http://localhost:8080` 即可查看资源导航页面。如需自定义端口或绑定域名，可修改启动命令参数。

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增优质资源链接、修复文档错误、改进页面样式或提交自动化脚本优化。

1.  **Fork 仓库并创建特性分支**：从 `main` 分支切出新分支，分支命名建议采用 `feat/batch-XX` 或 `fix/doc-typo` 格式，便于追溯变更意图。

2.  **更新数据文件**：若为新增资源，请在 `data/` 目录下找到对应的批次 JSON 文件，按 `{"url": "原始链接", "note": "可选备注"}` 格式追加条目。若需新建批次，请参考 `batch_42.json` 的结构模板。

3.  **本地验证**：运行 `python3 scripts/validate_urls.py` 检查所有 URL 是否满足格式要求（无协议补全、无大小写变动、无结尾斜杠）。运行 `python3 -m http.server` 在本地预览页面渲染效果。

4.  **提交 Pull Request**：提交前请确保 Commit Message 遵循约定式提交规范（如 `feat: add batch 43 resources`）。PR 描述中请说明资源来源或推荐理由，便于维护者评估。

5.  **等待 CI 检查通过**：GitHub Actions 将自动运行校验脚本，若检查失败请根据日志修复后再次推送。合并后，资源将在下一轮静态页面构建中生效。

## 常见问题

**Q: 为什么 URL 列表中的链接不包含 `https://` 或 `http://` 前缀？浏览器直接打开会报错吗？**

A: 项目设计理念是“所见即所得”——所有 URL 均忠实还原用户提交时的原始格式。部分裸域名（如 `www.bearfilm.xyz`）在大多数现代浏览器地址栏中会自动补全 `https://` 前缀并尝试访问。若需强制协议，建议用户在浏览器书签或自定义脚本中自行添加。项目本身不进行任何改写，以确保链接的可追溯性和原始意图。

**Q: 如果某个外链失效了，项目会如何处理？**

A: 项目作为静态索引，不主动探测链接可用性。维护者会定期（每季度）手动抽查高频访问链接，或依赖社区贡献者在 Issue 中报告死链。标记为失效的链接会被移至 `data/backups/` 下的归档文件，并从活跃索引中移除，但不会删除原始记录，以便日后回溯。

**Q: 能否将此项目部署到 GitHub Pages 或 Cloudflare Pages 上？**

A: 完全支持。只需将 `public/` 目录下的静态文件作为站点根目录上传即可。由于项目不包含任何服务端逻辑或构建步骤，任何支持静态托管的平台均可直接部署。建议在部署时开启强制 HTTPS，但请注意这不会影响项目内部对 URL 的原始呈现。
