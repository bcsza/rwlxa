# LinkPilot

LinkPilot 是一个面向技术内容创作者、开源项目维护者以及互联网资源管理者的轻量级外链资源聚合与导航系统。该项目定位于解决个人或团队在多个平台分散维护技术参考链接、项目依赖文档、视频教程入口以及社区推广页面时产生的管理混乱与访问效率低下问题。LinkPilot 不生产内容，但提供一套严谨、可扩展的目录结构与展示规范，帮助用户将零散的 URL 资源按照业务场景与技术层级进行归类、注解和统一呈现，最终生成一个可直接部署的静态导航站点或项目内嵌资源中心。

项目目标用户包括：需要维护多版本技术文档索引的 DevOps 工程师、运营多个垂直领域内容矩阵的数字媒体团队、以及希望为开源社区提供一站式外链跳板的项目发起人。LinkPilot 本身不依赖数据库或后端服务，完全基于 Markdown 与 YAML 前端描述数据驱动，支持一键导出为 JSON 或 HTML 静态页面，便于集成到现有 CI/CD 流程或嵌入到 GitHub Pages、Gitee Pages 等托管平台。通过标准化的资源条目模板，LinkPilot 强制要求每条外链附带所属类别、目标受众、更新频率和可靠性评分，从而在根源上提升资源聚合质量，避免链接失效或内容过时。

## 功能概览

**多级分类树管理** 支持用户自定义一级分类与二级子分类，每个分类可独立配置图标、描述与排序权重，便于构建从通用技术到垂直领域的完整导航路径。

**资源条目模板化** 提供预定义的 YAML 模板字段，包括 URL、标题、简短摘要、维护人、最后检查日期、状态标签（有效/失效/待审），确保每条外链信息完整且可追溯。

**自动链接健康检查** 集成可选的定时巡检脚本，基于 HTTP 状态码与响应时间对已收录链接进行可用性验证，生成异常报告并标记失效条目。

**多种输出格式适配** 内置渲染引擎支持输出为 Markdown 目录表格、HTML 响应式导航页、JSON API 数据接口，满足静态站点生成器、单页应用或命令行工具的不同消费场景。

**标签与全文检索** 为每条资源分配多个文本标签，并基于 MiniSearch 库提供轻量级前端模糊搜索能力，支持按标题、摘要、分类、标签组合过滤。

**权限与审核工作流** 针对多人协作场景，提供简单的状态机管理（草稿/待审/已发布/已下架），并可生成变更日志，方便团队回溯修改历史。

## 应用场景

技术博客外链整理 技术博主在撰写系列教程时，需要引用大量外部文档、API 参考和工具站点。LinkPilot 可为每篇文章生成独立的资源附录，并统一维护在项目仓库中，避免文章内嵌链接散落各处，同时便于读者一键访问所有参考资料。

开源项目社区导航 开源项目维护者通常需要在 README 中列举贡献指南、行为准则、讨论论坛、问题追踪、CI 状态等众多外链。LinkPilot 提供标准化的外链区块，可直接嵌入项目根目录的文档体系，使社区成员快速定位所需入口，减轻维护者手动更新链接的负担。

视频课程配套资源库 在线教育机构或独立讲师制作技术视频课程时，常需提供随堂代码仓库、练习平台、扩展阅读列表。LinkPilot 允许按课程章节建立资源分组，并支持导出为 HTML 页面挂载到课程站点侧边栏，提升学员学习体验。

团队内部技术雷达 企业技术团队可借助 LinkPilot 搭建内部技术栈资源地图，收录常用中间件官网、内部 Wiki、监控面板、日志查询工具等，并利用健康检查功能定期验证内部服务的可达性，降低新人上手时的信息找寻成本。

## 快速开始

以下指令适用于 Linux/macOS 及 Windows WSL 环境，要求已安装 Git 与 Node.js 18 以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkpilot.git

# 进入项目目录并安装依赖
cd linkpilot
npm install

# 复制示例资源数据并启动开发服务器
cp config/links.example.yaml config/links.yaml
npm run dev
```

执行完毕后，访问控制台输出的本地地址（通常为 http://localhost:3000）即可查看默认资源导航界面。如需自定义资源列表，直接编辑 config/links.yaml 文件并保存，开发服务器将自动热重载。

第一，查阅现有 Issue 与 Project Board，确认您想解决的问题或希望添加的功能尚未被他人认领。若为新增资源分类或批量导入链接，建议先通过 Issue 讨论分类的命名规范与范围边界，避免后续重复劳动。

## 常见问题

Q: links.yaml 中的链接数量过多时，构建速度会明显下降吗？
A: 对于 2000 条以内的资源条目，构建时间通常在 3 秒以内。超过 5000 条时，建议启用增量构建模式（通过 --incremental 标志），该模式仅重新解析修改时间变化的条目。此外，健康检查模块默认采用并发请求池（最大并发 10），不会阻塞主构建流程。如果仍遇到性能瓶颈，可以考虑将链接数据拆分为多个 YAML 文件并通过 include 语法组合。

Q: 如何确保内部网络环境下的链接健康检查不泄露敏感信息？
A: 健康检查模块仅发送标准的 HTTP HEAD 请求，不携带任何 Cookie 或 Authorization 头。若需要检查内网服务（如内部 GitLab、Jenkins），请在 config/links.yaml 中为对应条目设置 internal: true 标记，并配置 checker.allowInternalHosts 白名单。项目不会记录任何请求负载或响应体，仅存储状态码和响应时间。对于高度敏感的内网地址，建议直接跳过自动化检查，改为手动维护 lastChecked 字段。

Q: 输出的 HTML 页面能否完全离线使用？
A: 可以。使用 npm run build -- --offline 命令生成的静态页面会将所有 CSS、JavaScript 和字体资源内联或打包为相对路径引用，无需 CDN 网络。但请注意，页面中收录的外部链接本身仍需网络访问才能跳转。离线模式下的搜索功能依赖预先生成的索引 JSON 文件，该文件会在构建时一同写入 dist 目录，因此用户可在无网络环境下正常浏览本地资源树并执行全文检索。
