# ResourceLink Core

ResourceLink Core 是一个面向技术团队与独立开发者的开源外链资产整合与导航工具。该项目并非传统意义上的网址书签管理器，而是一套基于静态站点生成范式的外部资源结构化编排与呈现框架。其核心定位在于帮助用户将分散于各类社区、文档站、私有仓库与第三方服务平台中的高价值外部链接，以可维护、可版本控制、可自动化部署的方式进行统一归集与展示。

ResourceLink Core 主要解决以下问题：技术团队在项目推进过程中，常面临文档散落、依赖地址变更、第三方平台入口遗忘等困扰，导致新成员上手成本高、运维排查链路长。本项目通过提供一套标准化的资源描述与渲染方案，使得所有外链资源能够像代码一样被管理、审查与迭代，最终输出为可供内部团队或公开社区访问的静态导航页面。项目本身不存储任何用户数据，仅提供链接编排能力，所有资源引用均以原始地址直链形式存在，确保信息传递的准确性与即时性。

## 功能概览

- **多源链接归集**：支持将不同平台、不同格式的外部链接统一纳入单一管理仓库，避免多平台切换导致的信息碎片化。

- **分类与标签体系**：内置灵活的分类索引机制，允许用户为每条链接赋予业务域、技术栈、使用频率等多元标签，便于后续检索与统计分析。

- **静态站点生成**：基于配置化的链接清单，自动生成结构清晰、响应式布局的静态 HTML 页面，无需动态后端服务即可部署至任意 Web 服务器或对象存储。

- **版本化变更追踪**：所有链接增删改操作均通过文本配置文件完成，配合 Git 等版本控制系统，可完整追溯每一次资源变更的上下文与责任人。

- **健康状态探测**：提供可选的链接可达性预检能力，在构建阶段自动识别失效或响应异常的外部地址，并生成警告日志，辅助维护者及时清理或更新。

- **权限分级视图**：支持通过环境变量或构建参数控制输出页面的可见范围，例如区分内部测试环境与生产公开环境，不同环境展示不同的链接集合。

- **导入导出适配器**：提供 JSON、YAML、CSV 等多种格式的链接数据导入导出接口，方便与现有资产管理系统或电子表格工具进行数据交换。

## 应用场景

- **技术文档站外链整合**：团队在编写技术文档或 API 手册时，经常需要引用外部规范、参考实现或依赖包主页。使用 ResourceLink Core 可将这些引用统一维护，并在文档构建时自动注入，确保引用地址的一致性与可更新性。

- **微服务架构依赖拓扑展示**：在微服务治理场景中，各服务间存在错综复杂的调用关系与配置中心地址。本项目可被用作服务依赖关系的导航入口，将注册中心、配置仓库、监控面板、日志索引等外部地址按服务维度组织，降低运维人员查找关键入口的时间成本。

- **开源项目社区资源导航**：开源项目维护者通常需要维护贡献指南、行为准则、社区论坛、会议录屏、周边工具等大量外部链接。利用本项目，维护者可快速生成一份结构化的社区资源导航页，并随项目代码一同版本发布。

- **内部工具链统一入口**：企业内部往往同时运行着 Jenkins、SonarQube、Nexus、Harbor、GitLab 等多个基础工具平台。通过本项目，可将这些平台的访问地址与使用说明整合为单一入口页面，新入职员工可一键获取所有开发运维工具的位置与用途。

## 快速开始

以下指令演示如何从零开始获取 ResourceLink Core 源码、安装依赖并启动本地开发服务器。

```bash
# 克隆项目仓库至本地
git clone https://github.com/resourcelink/core.git

# 进入项目根目录
cd core

# 安装项目依赖（基于 npm）
npm install

# 启动开发模式，默认监听本地 3000 端口
npm run dev
```

执行上述命令后，访问本地开发服务器地址即可预览默认的链接导航页面。如需自定义链接数据，请编辑项目根目录下的 `config/links.yaml` 文件，修改后保存，开发服务器将自动热重载并刷新页面内容。

## 常见问题

**问：配置文件中的链接地址发生变化时，是否需要重新部署整个站点？**

答：是的。ResourceLink Core 采用静态生成模式，所有链接信息在构建时被编译进最终的 HTML 页面中。当您修改 `config/links.yaml` 文件中的任何地址后，需要重新执行构建命令（如 `npm run build`）并部署生成的 `dist` 目录内容，变更才会在线上生效。我们推荐将构建与部署流程集成至持续集成流水线中，以实现配置变更后的自动化重新部署。

**问：项目是否支持对链接进行分组或设置访问权限，例如仅内部网络可见的部分链接？**

答：项目本身不提供服务端鉴权能力，但通过构建时的环境变量机制，您可以实现环境级别的链接区分。具体做法是在 `config/links.yaml` 中使用环境变量占位符（如 `${ENV}`），然后在构建时通过设置 `ENV=internal` 或 `ENV=public` 等不同环境值，生成包含不同链接集合的静态页面。您也可以将生成的静态文件部署至具有访问控制策略的 Web 服务器或云存储服务中，以实现网络层面的权限隔离。

**问：如何验证配置文件中引用的外部链接是否仍然有效？**

答：项目提供了内置的链接健康预检脚本，您可以在构建前手动执行 `npm run check-links` 命令。该脚本会遍历配置文件中的所有 HTTP/HTTPS 链接，发送 HEAD 请求以验证其可达性，并在控制台输出检查结果与状态码。若检测到大量失效链接，构建过程会输出警告但不会中断，您可以依据警告日志进行针对性修复。请注意，该功能依赖网络环境，且对于部分需要特定请求头或会话验证的链接可能无法准确判断。
