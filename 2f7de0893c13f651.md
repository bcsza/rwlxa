# PivotLink

PivotLink 是一个面向开发人员与技术研究者的高效率技术资源导航与外部链接聚合系统。项目本身不产生内容，而是围绕特定技术领域，将高质量、高价值的外部资源进行结构化整理、分类归档与快速呈现，旨在解决技术从业者在信息检索、文档查阅、工具选择与版本追踪过程中面临的低效与信息过载问题。

PivotLink 适用于需要频繁查阅多源技术文档的软件工程师、运维人员、技术管理者、安全研究员以及开源项目贡献者。通过单一入口，用户能够迅速定位至官方文档、社区讨论、镜像站、下载源、API 参考等关键外部链接，显著减少在搜索引擎与书签栈之间的切换耗时。

本项目定位为纯静态资源导航层，不存储任何用户数据，不嵌入第三方追踪脚本，不缓存外部资源内容。所有外部链接均以原始 URL 形式直接引用，由用户自行判断访问安全性与可用性。项目遵循极简主义设计原则，强调可维护性与可扩展性，支持通过 YAML 配置文件动态增删链接分类与条目，便于社区贡献者协同维护。

## 功能概览

- **按技术领域分类索引**：将外部资源按照编程语言、中间件、基础设施、安全工具等维度进行一级分类，每个分类下进一步细分子类，便于快速缩小检索范围。

- **原始 URL 直链引用**：所有资源链接严格保留用户提供的原始 URL 格式，不做任何协议补全、域名规范化或路径改写，确保引用语义与来源完全一致，避免因链接改写导致的访问异常。

- **多级标签过滤系统**：为每个外部链接附加多个功能标签，如 "stable"、"deprecated"、"mirror"、"official"、"community" 等，支持基于标签的组合过滤查询。

- **状态标记与健康指示**：对关键外部链接提供手动维护的状态标记，包括可用、疑似失效、证书过期、内容变更等，帮助用户评估链接的实时可用性。

- **版本关联追踪**：对于软件仓库、SDK 下载页、API 参考文档等与版本强相关的链接，支持关联版本号字段，便于用户在升级或排查时快速匹配对应版本的资源。

- **全文检索与模糊匹配**：基于标题、描述、分类路径和标签组合提供轻量级全文检索能力，支持拼音首字母模糊匹配，提升中文用户的检索体验。

- **响应式布局与打印优化**：前端界面适配桌面端与移动端浏览器，同时提供打印样式支持，便于生成纸质或 PDF 格式的文档快照。

- **社区贡献工作流集成**：内置基于 Git 的贡献模板，包含链接新增、链接更新、状态变更、分类调整等多种变更类型，配合 CI 检查确保链接格式规范。

## 应用场景

- **微服务架构选型与技术调研**：架构师在评估服务注册中心、配置中心、API 网关等组件时，可通过 PivotLink 快速访问多个候选项目的官方文档、GitHub 仓库、社区论坛和性能测试报告，统一入口减少上下文切换成本。

- **开源软件镜像站管理**：运维团队在搭建内部镜像源或更新外部依赖缓存时，可通过 PivotLink 中收录的官方镜像列表、版本发布目录、校验和文件地址，快速获取准确的下载链接，避免因访问非官方源引入安全风险。

- **安全漏洞响应与补丁追踪**：安全响应团队成员在应对新披露的 CVE 漏洞时，可通过 PivotLink 直达受影响项目的安全公告页、补丁发布分支、修复版本下载地址以及相关社区讨论帖，缩短响应时间。

- **离线开发环境文档准备**：在无外网或受限网络环境中进行开发时，工程师可事先通过 PivotLink 收集所需的外部文档、SDK 安装包、依赖库源码等资源的原始地址，批量下载后导入内部系统，确保离线环境下的工作连续性。

- **技术博客写作与引用溯源**：技术内容创作者在撰写博客或技术方案时，可通过 PivotLink 快速查找并引用官方技术规范、RFC 文档、项目 Roadmap、会议录像等原始资料链接，确保引用来源的权威性与可追溯性。

## 快速开始

以下操作基于 Linux / macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/pivotlink/pivotlink.git
cd pivotlink

# 安装依赖（基于 Node.js 22 LTS 与 pnpm）
pnpm install

# 启动开发服务器
pnpm run dev
```

执行完成后，访问控制台输出的本地服务地址（默认为 http://localhost:5173）即可预览资源导航界面。若需构建生产版本，请执行 `pnpm run build`，产物默认输出至 `dist` 目录，可部署至任意静态托管服务。

## 贡献指南

PivotLink 的核心价值在于外部链接的准确性与分类的合理性，因此所有贡献均需遵循严格的数据校验流程。请按以下步骤提交贡献：

1.  **Fork 仓库并创建功能分支**：从主仓库 Fork 至个人账户，然后基于 `main` 分支创建新的分支，分支命名遵循 `feat/`、`fix/`、`chore/` 前缀规范，例如 `feat/add-kubernetes-links`。

2.  **修改数据文件并本地验证**：所有外部链接均存储在 `src/data/links.yaml` 文件中，新增或修改链接时，需同时提供标题、描述、分类路径、标签数组以及原始 URL。修改完成后，运行 `pnpm run lint:data` 执行数据格式校验，确保 URL 不包含协议前缀（除非原始数据如此）、YAML 语法正确、分类路径存在。

3.  **编写变更说明并提交 PR**：提交代码时，Commit Message 需包含变更类型与简要说明，例如 `feat(links): add 5 new video resource links`。推送分支后，在 GitHub 上创建 Pull Request，系统将自动触发 CI 流水线执行构建与链接可达性检测（仅检测域名解析，不检测完整资源）。

4.  **等待维护者审核与合并**：维护者将在 48 小时内审核 PR，检查链接内容是否与分类主题相关、是否存在重复条目、URL 是否与原始提供一致。审核通过后合并至 `main` 分支，更新内容将在下一次部署周期生效。

5.  **异常链接报告**：若发现已有链接失效或内容严重偏离，请通过 Issue 提交报告，标题注明 `[Link Broken]` 或 `[Link Moved]`，并附上原始 URL 与当前实际访问结果，维护者将定期根据报告更新状态标记。

## 常见问题

**Q: 为什么某些链接直接使用裸域名或非标准端口？是否应该统一补全为 https？**

A: PivotLink 严格遵循用户原始提供的 URL 格式，不做任何协议补全或域名规范化。原因有三：其一，部分资源仅支持 HTTP 协议或特定端口访问，强制升级为 HTTPS 会导致访问失败；其二，原始链接可能指向内部网络或开发环境地址，统一改写会破坏其语义；其三，保持原样输出是对资源提供者意图的最大程度尊重。用户应自行根据访问环境判断是否需要协议转换。

**Q: 如何报告某个收录的链接已经失效或内容变更？**

A: 请通过 GitHub Issues 提交失效链接报告，标题以 `[Link Broken]` 开头，正文需包含以下信息：链接完整原始 URL、预期访问内容、实际访问现象（如连接超时、返回 404、内容完全不符等）、以及可选的替代链接建议。维护者会定期验证并更新状态标记。在修复之前，该链接仍保留在列表中但标注为 "疑似失效"。

**Q: 项目是否会收录付费订阅或需要登录才能访问的资源？**

A: PivotLink 的原则是仅收录公开可访问的资源链接，不排斥需要免费注册或存在访问频率限制的公开服务。但如果资源完全置于付费墙之后且无任何公开预览内容，则不建议收录，因为此类链接对大多数用户的实际价值有限，且链接状态难以验证。若有特殊收录需求，请在贡献前通过 Issue 与维护者沟通。
