# LinkMatrix

LinkMatrix 是一个面向技术内容创作者、资源整理爱好者与信息聚合站运营者的开源外链管理与资源导航解决方案。本项目并非一个传统意义上的内容管理系统，而是一套以纯静态 Markdown 为核心的资源索引框架，旨在帮助用户以极低的维护成本构建结构清晰、可长期演进的个人或团队资源导航站。

项目目标用户包括：希望将散落于各处的优质外链进行系统化归档的技术博主、需要为内部团队提供统一入口的研发小组负责人、以及希望以公开透明方式运营垂直领域资源列表的社区维护者。LinkMatrix 不提供数据库、不依赖后端服务，所有资源记录均通过 Markdown 表格与列表进行组织，配合本项目提供的标准化目录结构与自动化校验脚本，用户可以快速生成可部署为静态站点的资源索引库。

LinkMatrix 解决的核心问题是：在信息过载的环境下，如何用最轻量的方式维持一份高可用、可审查、可协作的外链清单。通过强制性的 URL 原样记录规则与层级分明的目录设计，本项目帮助用户避免链接失效、格式混乱与维护懈怠等常见问题，使资源列表真正成为长期可用的知识资产。

## 功能概览

**原样 URL 强制记录机制** 系统内置链接格式校验脚本，确保每条外链均按照用户输入的原始形式完整保存，禁止自动补全协议头或域名前缀，杜绝因格式篡改导致的链接访问异常。

**分级目录自动生成** 根据资源主题与使用频率，项目脚本可自动生成多级分类目录结构，用户仅需将新链接填入对应类别的 Markdown 文件，即可完成站点内容更新。

**静态站点渲染支持** 提供与流行静态站点生成器（如 Hugo、VuePress）的适配模板，用户执行单条构建命令即可将整个资源目录渲染为可供公网访问的 HTML 站点。

**链接状态周期性检测** 内置基于 Shell 与 curl 的链接可达性检测脚本，可定期输出失效链接报告，辅助维护者及时清理或更新已变更的资源地址。

**协作审核工作流** 支持基于 Git 分支的贡献者提交模型，每位贡献者新增或修改链接时需创建独立分支，通过预设的审核清单后方可合并至主分支，确保资源质量。

**多格式数据导出** 支持将资源列表导出为 JSON、CSV 及纯文本三种格式，便于用户将数据迁移至其他平台或进行二次加工分析。

## 应用场景

个人技术博客的外链增补页 技术博主可在博客侧栏固定位置放置 LinkMatrix 生成的资源导航页链接，用于整理日常阅读中发现的优质开发工具、学习资料与社区讨论区，避免优质内容散落在浏览器书签中无从查找。

团队内部开发环境统一入口 研发小组可使用 LinkMatrix 搭建内部私有资源列表，集中存放测试环境地址、代码仓库镜像、CI/CD 控制台、日志查询面板等常用内网链接，新成员入职时仅需获取一份静态页面即可快速熟悉团队基础设施。

垂直领域公开资源聚合站 开源社区维护者可围绕特定技术主题（如 Rust 生态工具链、前端构建工具演进、数据库性能调优案例）构建公开资源导航站，通过 Pull Request 方式接受社区贡献，形成领域知识共同维护的良性循环。

离线文档配套资源索引 对于需要分发离线技术文档包的项目，可将 LinkMatrix 生成的资源列表作为文档包的附录页面，为读者提供延伸阅读与相关项目跳转指引，弥补离线文档在动态链接方面的不足。

## 快速开始

执行以下命令可在 3 分钟内完成 LinkMatrix 项目的本地克隆、依赖安装与初始站点运行：

```bash
git clone https://github.com/linkmatrix/linkmatrix-starter.git my-resource-hub
cd my-resource-hub
chmod +x scripts/setup.sh && ./scripts/setup.sh
make serve
```

执行完毕后，本地服务将启动于 http://localhost:8080，用户可通过浏览器访问初始示例资源列表页面。如需自定义资源内容，可直接编辑 `content/` 目录下的 Markdown 文件，保存后刷新页面即可看到更新。

## 贡献指南

贡献者请遵循以下步骤参与 LinkMatrix 资源列表的维护与更新：

第一，Fork 本项目仓库至个人账号，并在本地 clone 已 fork 的副本。请确保本地 Git 配置中 user.name 与 user.email 为有效值，以便提交记录可追溯。

第二，新建独立功能分支，分支命名建议采用 `add-resource-<分类>-<简述>` 或 `fix-link-<失效链接域名>` 格式。所有新增或修改操作均应在分支中完成，禁止直接向主分支提交。

第三，编辑对应分类的 Markdown 文件时，请严格遵守 URL 原样记录规则：复制用户提供的原始地址时不得添加 `http://` 或 `https://` 前缀，不得改动域名大小写，不得在末尾追加斜杠。新增链接后需运行 `make check` 执行格式校验，确保通过后方可继续。

第四，提交变更并推送至个人远程仓库，随后在原始项目页面发起 Pull Request。PR 描述中请注明新增资源的简要用途说明或修改原因，便于维护者进行审核。

第五，等待维护者审核。审核通过后，您的贡献将被合并至主分支，并在下一轮站点构建中自动生效。若 PR 状态超过 5 个工作日未有响应，可在 PR 评论区以礼貌方式提醒维护者。

## 常见问题

问：为什么规则强制要求 URL 原样输出，不允许自动补全协议头？

答：因为部分资源站域名可能同时存在 HTTP 与 HTTPS 服务，且两者内容或证书状态不一致。自动补全协议可能导致用户无法访问实际可用的版本。原样记录确保访问行为与贡献者测试时完全一致，避免因平台自动改写造成的访问失败。此外，部分内网或测试环境域名不支持 HTTPS，强制补全会直接导致访问中断。

问：链接检测脚本报告某条链接不可达，但我确认浏览器中可以打开，应该如何处理？

答：这种情况通常是因为目标站点对非浏览器 User-Agent 或自动化请求做了限制。建议先手动在浏览器中验证链接有效性，若确认可用，可在链接所在行后添加注释 `<!-- skip-check -->`，检测脚本将跳过该条链接的自动检测。同时请在贡献说明或 PR 中备注该情况，便于其他维护者知悉。

问：能否将本项目用于商业闭源项目内部资源管理？

答：可以。本项目采用 MIT 许可证，对使用场景不做任何限制，包括商业用途、闭源部署或内部私有化改造。您无需公开您修改后的源码，也无需在衍生项目中保留完整的项目声明，但建议保留原始许可证副本以示对开源社区的基本尊重。
