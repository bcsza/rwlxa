# CloudLink Navigator

CloudLink Navigator 是一个轻量级、可自托管的开源技术资源导航与外链聚合平台。它面向开发团队、技术社区及个人研究者，旨在解决分散的网络资源管理、团队知识共享及快速访问入口构建等问题。通过结构化的目录分类与简洁的展示逻辑，该项目将零散的优质外部链接转化为组织内部可复用、可维护的知识资产。

## 功能概览

- **多级分类目录管理**：支持管理员自定义无限层级的分类体系，便于按技术栈、业务领域或使用频率对资源进行逻辑分组，满足不同规模团队的梳理习惯。
- **外部链接安全跳转**：所有外链均经过中间页过滤，可配置跳转提示或风险校验，降低因直接访问未知站点带来的安全风险，同时便于统计出站流量。
- **全文检索与标签筛选**：内置基于标题、描述及标签的轻量级全文搜索，配合标签云快速缩小范围，帮助用户在大量链接中精准定位所需资源。
- **响应式前端界面**：基于 CSS Grid 与 Flexbox 布局，适配桌面、平板及移动设备，确保在不同屏幕尺寸下均有良好的浏览与操作体验。
- **RESTful API 支持**：提供完整的 JSON 格式数据接口，方便与现有监控系统、自动化脚本或第三方应用集成，实现资源数据的同步与二次处理。
- **访问热度统计看板**：内置简易计数器，记录每个链接的点击次数，并在管理后台提供排序视图，辅助判断资源价值与团队关注趋势。
- **数据导入导出工具**：支持 CSV 与 JSON 格式的批量导入及定期备份导出，降低迁移成本，便于与其他书签管理工具或文档系统交换数据。

## 应用场景

- **技术团队内部知识库**：开发团队可将日常使用的 API 文档、设计规范、代码仓库及运维面板统一收录，形成团队专属的起始页，新人入职即可快速上手，减少询问与查找耗时。
- **开源项目生态导航**：开源项目维护者可利用该平台聚合周边的社区论坛、贡献指南、示例代码库及扩展插件列表，帮助贡献者与使用者更全面地了解项目周边生态。
- **个人技术信息中枢**：技术爱好者或研究者可搭建个人实例，将日常跟踪的博客、周刊、技术播客及在线工具集中管理，配合标签系统构建个人化的学习路线与信息过滤通道。
- **企业内部门户辅助**：作为企业内部导航站的有力补充，用于承载非核心但常用的业务系统链接（如日志平台、监控告警、测试环境入口），减轻主门户的信息过载压力。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，请确保已安装 Git 与 Node.js（v18+）。

```bash
# 克隆项目仓库
git clone https://github.com/cloudlink-navigator/core.git
cd core

# 安装项目依赖
npm install

# 启动开发服务器（默认监听 3000 端口）
npm run dev
```

访问控制台输出的本地地址（通常为 http://localhost:3000）即可开始使用。生产环境部署请参考 `docs/deployment.md` 中的 Nginx 或 Docker 配置示例。

## 贡献指南

1.  **查阅议题与计划**：访问 GitHub Issues 页面，了解当前已知问题与 roadmap 计划。对于较大功能改动，建议先创建新议题进行讨论，避免无效开发。
2.  **派生仓库并创建分支**：将项目派生至个人账户，基于 `main` 分支创建以 `feature/` 或 `fix/` 为前缀的命名分支，例如 `feature/enhance-search-filter`。
3.  **遵循编码规范**：JavaScript 代码须通过 ESLint 配置（参见 `.eslintrc.js`）的校验，提交前执行 `npm run lint` 自动修复部分格式问题。提交信息须采用 Conventional Commits 风格。
4.  **编写测试用例**：新增功能或修复缺陷时，应在 `tests/` 目录下补充对应的单元测试或集成测试，确保 `npm test` 全部通过。
5.  **提交拉取请求**：推送分支后，向主仓库的 `main` 分支发起 Pull Request，并在描述中清晰关联相关议题、说明改动要点及测试覆盖情况。所有 PR 须经过至少一位维护者审阅。

## 常见问题

**问：项目是否支持 PostgreSQL 替代默认的 SQLite？**

答：支持。项目使用 Knex.js 作为查询构建器，已在配置中预留 PostgreSQL 驱动。您只需修改 `config/production.json` 中的 `client` 与 `connection` 字段，并安装 `pg` 依赖包，重启服务后即可自动迁移数据表结构。详细步骤参见 `docs/administration.md`。

**问：如何在私有网络环境下离线部署？**

答：对于完全离线的环境，您需要在有网络的机器上下载全部依赖包（`npm pack` 或使用 `npm-offline` 工具），并将其传输至目标服务器。同时，需将 `config/default.json` 中的 `offlineMode` 设为 `true`，以禁用 CDN 资源加载及外部更新检查。前端资源已全部内联，不依赖外部网络。

**问：统计点击数是否会显著影响性能？**

答：每一次点击均采用异步写入方式，并经过内存队列缓冲，每 10 秒批量落盘，不会阻塞主请求响应路径。在默认 SQLite 配置下，可支撑每秒约 200 次点击记录写入，对于大多数中小规模团队完全够用。如需更高并发，建议切换至 PostgreSQL 并调整连接池大小。
