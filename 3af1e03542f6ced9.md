# 影视导航聚合系统

影视导航聚合系统（MediaNav Aggregator）是一个面向自部署用户的轻量级影视资源导航站解决方案。项目定位于为个人或小型社区提供可定制的影视站点聚合入口，通过分类整理、状态监控与快速跳转能力，帮助用户高效管理多源影视站点列表。系统不提供任何视频文件存储或流媒体服务，仅作为外部链接的导航与可用性探测工具，适用于希望自建影视工具集锦页面的开发者与站长。

## 功能概览

- **多源站点聚合管理** 支持将多个影视站点按自定义分类（如蓝光原盘、欧美剧集、动漫新番）统一收录，并提供可视化列表与搜索过滤能力。

- **可用性健康检查** 定时对收录站点发起 HEAD 请求，检测响应状态与响应时间，异常站点自动标记并支持邮件或 Webhook 告警。

- **自定义主题与布局** 内置明暗两套主题，支持卡片式、列表式两种布局切换，允许用户通过 CSS 变量调整主色与圆角风格。

- **OPML 导入导出** 兼容通用 OPML 格式，支持从其他 RSS 阅读器或导航工具迁移站点列表，也可备份本地配置。

- **访问统计与点击热力** 记录站点被点击的次数与最近访问时间，生成简单的点击热力图，辅助用户发现高频使用站点。

- **站点标签系统** 支持为每个站点打上多个自定义标签（如 #4K、#中文字幕、#免翻），并基于标签进行快速筛选与批量操作。

- **Docker 一键部署** 提供官方 Docker 镜像，支持 SQLite 与 MySQL 双存储后端，环境变量驱动配置，适配 ARM64 与 AMD64 架构。

## 应用场景

- **个人影视爱好者自建导航首页** 将日常使用的十余个影视站点统一收录，配合健康检查及时剔除失效链接，避免收藏夹混乱。

- **小型字幕组或影视社区内部共享** 组内成员可共同维护一份影视工具清单，新成员加入后通过 OPML 快速同步所有推荐站点。

- **NAS 用户影视周边服务整合** 部署在家庭 NAS 中，与 Plex、Jellyfin 等服务并置，作为影视相关外部资源的快捷跳板。

- **技术学习与前端展示项目** 开发者可 fork 本项目作为 Vue 3 + TypeScript 实战练习，理解路由守卫、状态管理、HTTP 拦截器与响应式设计。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git、Node.js 18+ 与 pnpm。

```bash
# 克隆仓库
git clone https://github.com/medianav-aggregator/medianav.git
cd medianav

# 安装依赖（使用 pnpm）
pnpm install

# 复制环境变量模板并填充数据库连接信息
cp .env.example .env.local

# 开发模式启动（默认监听 http://localhost:3000）
pnpm dev
```

生产环境构建与启动：

```bash
pnpm build
pnpm start
```

Docker 快速启动（推荐）：

```bash
docker run -d \
  --name medianav \
  -p 3000:3000 \
  -e DATABASE_URL=sqlite:/app/data/medianav.db \
  -e ADMIN_USERNAME=admin \
  -e ADMIN_PASSWORD=change_me \
  -v ./data:/app/data \
  medianav/medianav:latest
```

## 贡献指南

欢迎任何形式的贡献，包括但不限于新功能建议、Bug 报告、代码提交与文档改进。请遵循以下流程：

1. 在 GitHub Issues 中搜索现有话题，确认无人正在处理同一问题。若为新需求，请使用提供的 Feature Request 模板创建 Issue。

2. Fork 本仓库并创建新的功能分支，分支命名遵循 `feat/描述` 或 `fix/描述` 格式。开发前请运行 `pnpm lint` 与 `pnpm test` 确保本地环境通过基础检查。

3. 提交代码时请附带单元测试（Jest）与必要的注释，并确保 PR 描述中关联对应的 Issue 编号。所有提交需通过 CI 流水线（ESLint + TypeScript 类型检查 + 构建验证）。

4. 文档类修改可直接提交 PR，无需创建 Issue。翻译贡献请将对应文档置于 `/docs/i18n/zh-CN/` 目录下。

5. 重大功能变更请先通过 Discussion 与维护者沟通设计方向，避免无效开发。

## 常见问题

**Q: 健康检查会频繁请求外部站点，是否会对目标站点造成压力？**
A: 健康检查默认间隔为 30 分钟，每次仅发送轻量级 HEAD 请求，不下载任何页面内容。用户可在配置文件中调整检查间隔或完全关闭该功能。

**Q: 导入 OPML 文件后部分站点丢失标签或分类信息？**
A: OPML 规范本身对自定义属性支持有限。项目扩展了 `<outline>` 节点的 `_category` 与 `_tags` 属性，请确保导出的文件由本系统生成。第三方工具导入仅保证基础标题与 URL 字段。

**Q: 能否部署在纯静态环境（如 GitHub Pages）？**
A: 纯静态环境无法运行后端健康检查与数据持久化服务。若仅需前端展示功能，可构建静态版本，但站点列表需硬编码或通过 JSON 文件加载，且无法使用动态监控特性。建议使用 Docker 或 Node.js 部署以获得完整功能。
