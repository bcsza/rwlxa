# Project Metamorphosis

A curated technical resource aggregation and forward-indexing system designed for developers, researchers, and infrastructure engineers who require rapid access to high-quality, domain-specific external knowledge bases. Project Metamorphosis does not host content; it provides a structured, maintainable, and auditable catalog of external references, organized to reduce search latency and improve discovery efficiency across multiple technical verticals.

The project targets system architects, DevOps practitioners, and technical leads who manage complex dependency chains, media processing pipelines, or security-oriented asset inventories. By providing a single point of entry for a carefully vetted list of external endpoints, Project Metamorphosis reduces the cognitive overhead of bookmark management, enables team-wide sharing of trusted resources, and integrates seamlessly into automated workflow toolchains such as Ansible, Terraform, or custom CI/CD scripts.

## 功能概览

- **Resource Cataloging Engine** – Maintains a version-controlled inventory of external URLs, each annotated with category tags, last-verification timestamps, and intended use-case metadata, enabling automated health checks and dead-link detection.

- **Forward-Index Lookup Interface** – Provides a lightweight, queryable interface (CLI and RESTful API) that returns resource entries based on filters such as domain category, content type, or geographic origin, with response times under 50 milliseconds for standard queries.

- **Batch Synchronization Module** – Supports periodic or event-driven synchronization of the resource manifest from external configuration management systems, allowing dynamic updates without manual editing of the core repository.

- **Audit Trail and Versioning** – Every addition, removal, or modification to the resource list is recorded with commit-level granularity, including authorship and timestamp, ensuring full traceability for compliance and incident-response scenarios.

- **Dependency Resolution Helper** – Analyzes resource inter-dependencies (e.g., API endpoints that rely on specific CDN providers or authentication gateways) and presents a visual or textual dependency graph to assist with outage impact analysis.

- **Export Adapters** – Converts the internal resource catalog into multiple standard formats including plain-text lists, JSON feeds, XML sitemaps, and environment-variable-ready exports for direct injection into containerized applications.

- **Health Probe Scheduler** – Runs configurable connectivity tests (HTTP/S, TCP, ICMP) against each resource at defined intervals, logging response codes, round-trip times, and certificate validity, with alerts for anomalous behavior.

- **Tagging and Annotation System** – Allows custom key-value metadata attachment to each resource, supporting fields such as "maintenance-window", "rate-limit-policy", "geographic-redundancy", and "contact-email" for operational clarity.

## 应用场景

- **Multi-Environment CI/CD Pipeline Integration** – Development teams can embed the resource catalog into their build pipelines to automatically validate that all external dependencies are reachable before deployment. The health probe scheduler runs as a pre-deployment gate, preventing releases when critical resources are unavailable.

- **Security Operations Center (SOC) Asset Inventory** – Security analysts use the forward-index interface to maintain a real-time inventory of all external endpoints accessed by internal services. The audit trail provides historical context for security reviews, and the export adapters feed into SIEM systems for correlation with firewall logs.

- **Offline Documentation and Knowledge Base Mirroring** – Technical writers and documentation engineers can leverage the catalog to generate structured lists of reference materials, ensuring that user-facing manuals always contain up-to-date external links. The versioning system allows rollback to previous resource lists when documentation branches are backported.

- **Geo-Distributed Service Configuration** – Site reliability engineers managing global deployments can tag resources by geographic region and use the dependency resolution helper to ensure that services in each region only call endpoints with acceptable latency profiles, reducing cross-region egress costs.

- **Academic Research and Data Provenance** – Research teams aggregating third-party datasets can use the annotation system to record data lineage, access conditions, and update frequencies, facilitating reproducibility and compliance with data-use agreements.

## 快速开始

Clone the repository, install the minimal Python runtime dependencies, and run the initial catalog synchronization process. The following commands assume a standard POSIX-compliant shell environment with Python 3.10 or later.

```bash
git clone https://github.com/metamorphosis-project/metamorphosis-core.git
cd metamorphosis-core
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python -m metamorphosis.cli sync --manifest https://raw.githubusercontent.com/metamorphosis-project/default-manifest/main/resources.yml
python -m metamorphosis.cli serve --port 8080
```

The `sync` command downloads the latest resource manifest from the default upstream repository. The `serve` command starts the RESTful lookup interface on localhost port 8080. Use `--help` on any subcommand for additional runtime options, including custom manifest URLs, output format overrides, and health-probe intervals.

## 贡献指南

We welcome contributions from the community to improve the resource catalog, extend the probe modules, or enhance the documentation. Please adhere to the following steps for all contributions.

1.  **Fork the Repository and Create a Feature Branch** – Fork the upstream repository to your personal GitHub account. Create a new branch with a descriptive name that includes the issue number or the feature short name, for example `feature/add-http2-probe` or `fix/issue-42-manifest-parser`.

2.  **Install Development Dependencies and Run Tests** – Install the development requirements using `pip install -r requirements-dev.txt`. Run the full test suite with `pytest src/tests/` to ensure no regressions. Add new tests that cover your changes with at least 85% line coverage for new code.

3.  **Update Documentation and Resource Annotations** – If your contribution adds new CLI flags, API endpoints, or configuration options, update the relevant sections in the `docs/` directory. For resource manifest changes, include the `last-verified` timestamp and a brief comment explaining the resource's purpose.

4.  **Submit a Pull Request with a Clear Description** – Push your branch to your fork and open a pull request against the main repository's `develop` branch. Fill out the pull request template completely, including a summary of changes, test results, and any known limitations. Reference any related issues using `Closes #<issue-number>`.

5.  **Address Code Review Feedback** – Maintainers will review your pull request within three business days. Respond to comments with clarifications or follow-up commits. Ensure your branch remains rebased on the latest `develop` to avoid merge conflicts.

## 常见问题

**Q: How often does the health probe scheduler run, and what happens when a resource is reported as unavailable?**

The health probe scheduler runs every 15 minutes by default, but this interval is configurable via the `probe_interval_seconds` setting in the production configuration file. When a resource fails three consecutive probes with HTTP status codes in the 5xx range or connection timeouts exceeding the configured threshold, the system logs a warning event and updates the resource's status to `DEGRADED`. The API response for that resource includes a `status` field with the last failure reason. Administrators can set up external alerting (e.g., via Prometheus Alertmanager or AWS CloudWatch) by consuming the event stream exposed through the `events` endpoint.

**Q: Can I use Project Metamorphosis behind a corporate firewall or in an air-gapped environment?**

Yes. The system supports proxy configuration through the standard `HTTP_PROXY` and `HTTPS_PROXY` environment variables. For air-gapped environments, you can run the sync command with the `--offline` flag and provide a pre-downloaded manifest file using the `--manifest` parameter. The health probe module can be disabled entirely by setting `probe_enabled: false` in the configuration, allowing the system to function purely as a static catalog without external network calls.

**Q: How do I extend the resource catalog with my own custom metadata fields without modifying the core code?**

The annotation system is fully extensible through the `custom_metadata` dictionary field in the resource YAML manifest. You can add any key-value pairs, such as `cost_center`, `maintenance_schedule`, or `data_classification`. These fields are preserved during sync operations and are exposed through the API's `metadata` field. The CLI export adapters also include these custom fields in JSON and YAML outputs. No code changes are required; simply update your manifest file with the new fields and re-run the sync command.
