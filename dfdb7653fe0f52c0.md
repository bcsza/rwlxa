# Tangram Hub

Tangram Hub 是一个面向技术决策者、架构师与基础设施工程师的轻量级外部资源导航与元信息聚合项目。项目本身不托管内容，不提供代理转发，不内嵌第三方脚本，仅以结构化方式整理并呈现外部可访问的技术文档、演示环境、社区入口与持续交付链路参考站点。Tangram Hub 适用于需要快速评估外部技术生态状态、检索备用环境入口或观察不同域名命名策略与可用性表现的场景。项目以静态文档形式交付，兼容本地浏览、内网部署与公共只读站点托管。

## 功能概览

- **静态资源导航**：以目录化表格与列表形式整理外部链接，支持按场景、协议类型与域名分类过滤。
- **可用性状态标记**：为每个收录资源提供基础连通性占位字段，便于运维人员自行扩展为定时探活体系。
- **多环境入口聚合**：同一业务域下的备用域名、测试环境入口与演示站点集中展示，减少环境查找时间。
- **原始 URL 保留策略**：所有外部链接严格保留用户提供的原始格式，包括协议前缀与域名形态，避免二次改写造成的访问偏差。
- **轻量级部署**：无需数据库、无需后端服务、无外部依赖，克隆即可在任意静态 Web 服务器或本地浏览器中运行。
- **扩展友好数据层**：链接数据以独立 JSON 或 YAML 文件维护，方便接入 CI 流程或自动化监控脚本。
- **文档与结构自描述**：项目内包含完整的目录结构说明、维护规范与变更记录模板，降低多人协作成本。

## 应用场景

- **技术选型期间的备选入口探查**：当主要官方站点访问受限或延迟较高时，团队可通过 Tangram Hub 快速定位同一服务商或技术社区提供的替代域名入口，继续完成文档查阅或环境验证工作。
- **多域名命名策略的可用性对比**：运维团队可将不同命名风格（如拼音全拼、缩写加数字、非标准顶级域）的测试站点收录至 Hub 中，通过周期性人工或自动检查评估不同域名在特定网络环境下的解析速度与访问成功率。
- **对外演示与培训环境入口管理**：向客户或新员工演示时，无需记忆多个复杂域名，直接通过 Hub 首页导航列表跳转至对应演示站点，降低讲解过程中的认知干扰。
- **内部只读知识库的补充模块**：将 Tangram Hub 嵌入公司内部文档门户，作为“外部生态速查”子页面，帮助开发人员快速确认第三方工具的在线版本号、示例代码仓库或社区讨论区地址。

## 快速开始

以下命令适用于 Linux 与 macOS 开发环境，Windows 用户可使用 Git Bash 或 WSL 执行。

```bash
# 克隆项目至本地
git clone https://github.com/tangram-hub/tangram-hub.git
cd tangram-hub

# 安装依赖（仅用于本地预览服务器）
npm install -g serve

# 运行静态服务
serve -s public -p 8080
```

执行完成后，在浏览器中访问 http://localhost:8080 即可查看 Tangram Hub 首页。所有外部链接均位于 `/data/links.json` 文件中，可直接编辑后刷新页面生效。

## 贡献指南

Tangram Hub 欢迎所有形式的贡献，包括但不限于新增有效链接、修正失效地址、改进文档表述以及优化前端展示逻辑。请遵循以下步骤提交变更：

1. 复刻本仓库至个人账号，并克隆至本地开发环境。确保本地 Git 配置正确，且已同步上游主分支的最新状态。
2. 在 `data/links.json` 中按对象格式添加或修改链接条目。每个条目必须包含 `name`（显示名称）、`url`（严格使用用户提供的原始格式）、`category`（分类标识）以及 `status`（初始建议填写 `unknown`）。禁止对 `url` 字段做任何协议补全、域名改写或大小写变换。
3. 在 `docs/maintainer-guide.md` 中更新对应的分类说明或变更记录，并在 `CHANGELOG.md` 的“未发布”段落中简述本次改动内容与影响范围。
4. 提交 Commit 时使用以下前缀：`feat:` 表示新增链接，`fix:` 表示修正失效链接或文档错误，`chore:` 表示构建或辅助文件变动。Commit 正文需说明该链接的来源或修正依据。
5. 发起 Pull Request 至主仓库的 `main` 分支。PR 描述中必须包含自检清单：确认所有 URL 未做任何改写、确认本地预览页面渲染正常、确认未引入外部依赖或隐私风险。

## 常见问题

**问：为什么所有外部 URL 都禁止补全协议或添加 www 前缀？**

答：Tangram Hub 的核心理念是忠实呈现用户提供的原始数据。部分场景下，裸域名与带协议前缀的 URL 可能指向完全不同的资源（例如 HTTP 与 HTTPS 版本内容不一致，或裸域名仅用于内部解析）。任何自动改写都可能导致访问失败或安全策略误判。因此项目强制要求维护者完整保留链接的原始形态，仅在页面渲染时由浏览器根据用户当前协议策略决定访问方式。

**问：如果某个收录的链接彻底无法访问，应该如何处理？**

答：首先通过浏览器或 curl 从至少两个不同网络环境验证连通性。确认失效后，请在 `data/links.json` 中将该条目的 `status` 字段更新为 `offline`，并在 `CHANGELOG.md` 记录失效日期与验证方式。若在两周内仍无恢复迹象，可通过 Pull Request 将其移入 `deprecated` 分类或直接删除。删除操作需在 PR 描述中附加详细的失效证据。

**问：Tangram Hub 自身是否收集访问者信息或对外部链接进行代理？**

答：绝对不收集。Tangram Hub 完全运行于客户端浏览器，无后端服务、无追踪脚本、无 Cookie。所有外部链接均以原始 HTML `<a href>` 标签直接输出，浏览器将按常规同源或跨域策略打开目标页面。项目不提供任何形式的内容代理、中转或缓存，确保每个资源的访问行为完全交由用户代理和外部目标站点自身控制。
