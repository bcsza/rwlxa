# OSW (Open Source Web Navigator)

OSW 是一个轻量级的技术资源导航与外部链接聚合工具，定位为开源社区维护的优质网络资源索引系统。本项目面向开发者、技术研究者及内容创作者，帮助用户从海量网络信息中快速定位高价值的技术文档、媒体资源与实用工具站点。OSW 不存储任何第三方内容，仅提供结构化链接索引与基础元数据描述，旨在解决信息分散、优质入口难以发现的问题，同时为个人或小团队提供一套可私有化部署的资源目录框架。

## 功能概览

**资源分类索引**：按媒体类型、技术领域、内容形式等维度对收录链接进行多级标签分类，支持用户自定义分类规则。

**站点信息快照**：为每个收录的 URL 自动抓取并缓存页面标题、描述关键词及图标，帮助用户在点击前快速了解目标内容。

**可用性健康检查**：定期对收录链接进行 HTTP 状态探测，自动标记访问异常或响应超时的站点，并在管理界面高亮提示。

**用户收藏与备注**：允许注册用户对任意链接添加个人标签、备注文本及星级评分，形成个人化的资源筛选视图。

**全文检索与过滤**：基于标题、描述、分类标签及用户备注内容提供多字段模糊搜索，并支持按状态、分类、添加时间等维度组合过滤。

**开放数据导出**：支持将链接索引数据导出为 JSON、CSV 或 OPML 格式，便于迁移至其他阅读器或导航工具，同时提供只读 API 接口。

## 应用场景

个人技术博主可利用 OSW 构建自己的外链推荐页，将常用开发文档、在线工具、学习视频站点统一收纳，并通过分类标签和备注功能记录各站点的使用心得与适用场景，避免重复搜索。

小型开发团队可将 OSW 作为内部知识库的补充模块，集中存放项目依赖的开源库主页、API 参考文档、设计规范站点及团队成员的公开仓库地址，新成员入职时可快速了解团队常用的技术生态入口。

内容聚合与推荐类应用可调用 OSW 的只读 API，将本项目的链接索引数据嵌入到自己的资讯面板或浏览器起始页中，从而复用社区维护的优质资源清单，降低自身内容采集与筛选成本。

教育机构或技术培训组织可利用 OSW 创建课程配套资源导航，将每门课程涉及的在线实验环境、视频教程、习题解答博客等链接分门别类，并定期通过健康检查功能验证实验环境的可用性，确保教学过程中外部资源稳定可访。

## 快速开始

以下指令适用于 Linux / macOS / WSL 环境，需提前安装 Git 与 Node.js 18.x 及以上版本。

```bash
# 克隆项目仓库
git clone https://github.com/osw-community/osw-navigator.git
cd osw-navigator

# 安装项目依赖
npm install

# 使用 SQLite 本地数据库启动开发服务（默认端口 3000）
npm run dev
```

启动后访问 `http://localhost:3000` 即可进入导航首页。首次启动会自动初始化数据库表结构并生成示例分类数据。若需使用 PostgreSQL 或 MySQL 作为生产数据库，请参考 `docs/deployment.md` 中的环境变量配置说明。

## 项目结构

```
osw-navigator/
├── src/
│   ├── api/                     # RESTful API 路由定义（v1 版本）
│   │   ├── links/               # 链接资源的 CRUD 与检索接口
│   │   ├── categories/          # 分类树的查询与更新接口
│   │   └── health/              # 健康检查状态查询与手动触发接口
│   ├── core/                    # 核心业务逻辑层
│   │   ├── crawler/             # 页面元数据抓取模块（含超时重试与 User-Agent 轮换）
│   │   ├── checker/             # 可用性探测调度器（基于 node-cron 定时任务）
│   │   └── indexer/             # 全文索引构建与查询适配器（适配 SQLite FTS5 / PostgreSQL tsvector）
│   ├── models/                  # 数据模型定义（Sequelize / TypeORM 实体）
│   │   ├── Link.ts              # 链接主表模型（含 URL、标题、描述、分类外键、添加时间等）
│   │   ├── Tag.ts               # 用户自定义标签模型（多对多关联链接）
│   │   └── HealthRecord.ts      # 历史健康检查记录（含响应时间、状态码、错误信息）
│   ├── services/                # 外部服务集成层
│   │   ├── cache/               # Redis 缓存服务封装（支持 get / set / del / ttl）
│   │   └── queue/               # 可选的消息队列适配（用于将抓取任务异步化）
│   └── ui/                      # 服务端渲染视图模板（EJS）及前端静态资源
│       ├── assets/              # CSS、JavaScript、图标文件
│       └── views/               # 页面模板（首页、分类详情、管理后台、关于页面）
├── config/                      # 环境配置文件（development.json / production.json / test.json）
├── migrations/                  # 数据库迁移脚本（支持 SQLite 与 PostgreSQL 双方言）
├── tests/                       # 单元测试与集成测试（Jest + Supertest）
│   ├── unit/                    # 核心模块单元测试
│   └── integration/             # API 端到端测试与数据库事务回滚测试
├── docker-compose.yml           # 完整生产栈编排（Node + PostgreSQL + Redis + Nginx）
├── Dockerfile                   # 多阶段构建镜像文件（基于 Alpine Linux）
├── package.json                 # 项目依赖、脚本命令与元信息
├── README.md                    # 项目概览与快速入门（当前文件）
└── LICENSE                      # MIT 许可证文本
```

3. 参与代码贡献：查阅 `docs/development/` 中的开发指南，从 `good-first-issue` 标签的 issue 入手。提交 PR 前请确保本地通过全部单元测试及 ESLint 检查，并补充对应的测试用例与文档更新。

4. 完善项目文档：若发现文档中的错误描述、过时配置示例或翻译不完整，可直接编辑 `docs/` 目录下的 Markdown 文件并提交 PR。文档变更无需经过复杂的功能测试流程，但需要保持与代码实际行为一致。

5. 报告问题或安全漏洞：请通过 Issues 报告功能缺陷、性能问题或 UI 显示异常。若涉及潜在安全风险（如 XSS 注入、权限绕过等），请发送邮件至维护组邮箱（见 `CODEOWNERS` 文件），避免在公开渠道提前披露。

## 常见问题

**问：OSW 是否会存储第三方站点的内容或用户上传的文件？**

答：不会。OSW 仅存储每个链接的元数据（标题、描述、分类、添加时间）以及健康检查返回的状态码与响应时间。系统不代理、不缓存、不转发任何第三方站点的实际内容。所有外部链接均以 `target="_blank" rel="noopener noreferrer"` 形式打开，用户离开 OSW 域名后的行为与本项目无关。

**问：如何从旧版本迁移数据，或者切换数据库类型？**

答：项目在 `migrations/` 目录下提供了完整的 schema 定义与迁移脚本。若需从 SQLite 迁移至 PostgreSQL，可先运行 `npm run db:export -- --format=json` 导出所有链接与分类数据为 JSON 文件，再修改 `config/production.json` 中的数据库连接字符串，最后执行 `npm run db:import -- --file=export.json` 完成导入。迁移前请务必备份原始数据库文件。

**问：健康检查功能是否会影响被探测站点的正常服务？**

答：健康检查模块默认采用低频率探测策略（每 24 小时一次），且并发请求数限制为 5 个，同时设置 10 秒超时与 3 次重试间隔。检查请求携带的 User-Agent 标识为 `OSW-HealthChecker/1.0`，便于目标站点识别并做访问控制。若您的站点不希望被 OSW 探测，可在 robots.txt 中禁止 `OSW-HealthChecker` 爬虫，或联系项目组将该域名加入探测黑名单。
