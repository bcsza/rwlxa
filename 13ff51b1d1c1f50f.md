# QFys Navigator

QFys Navigator 是一个面向技术研究人员与开源爱好者的轻量级外部资源导航与聚合系统。项目定位为技术外链的中转枢纽，不托管实际内容，仅提供结构化、可维护的链接索引与元信息标注能力，帮助用户从分散的站点中快速定位有效信息。

项目面向需要频繁查阅外部技术文档、跟踪多个独立站点更新、或维护个人技术书签体系的中高级开发者。其核心价值在于将零散的域名与资源入口统一收纳，并通过简单的本地工具链实现分类、检索与状态检查，避免浏览器书签混乱或依赖在线收藏服务的隐私风险。

## 功能概览

- **多源外链聚合** 支持将任意数量的外部 URL 按类别、批次或主题组织为扁平列表，便于集中管理。
- **原始链接直出** 所有收录的 URL 均保留用户提供的原始格式（包含协议、域名、路径等），不做任何自动补全或改写，确保跳转准确性。
- **静态索引生成** 基于 Markdown 构建项目文档与资源清单，无需数据库或后端服务，可直接托管于 GitHub Pages 或任何静态服务器。
- **本地状态检查** 提供辅助脚本，可对已收录的域名进行基础可达性探测（HTTP 状态码），辅助发现失效链接。
- **分类标签系统** 支持为每个资源条目添加自定义标签（如 `docs`、`forum`、`tool`），便于按主题过滤。
- **批次追踪记录** 内置批次元数据（如当前第 46/48 批），记录资源收录的批次与时间，方便增量更新。
- **纯文本可维护** 所有配置与资源列表存储于 Markdown 及 YAML 文件中，易于版本控制与协作编辑。
- **低依赖运行环境** 仅依赖 Python 3.8+ 及标准库，无第三方包强制要求，降低部署门槛。

## 应用场景

- **个人技术书签库整理** 开发者可将长期积累的零散技术博客、在线工具、API 文档等入口统一录入项目，通过分类和注释快速回忆用途，替代浏览器书签的混乱管理。
- **团队知识库外链附录** 技术团队在维护内部 Wiki 或 Onboarding 文档时，可将本项目的资源列表作为附录嵌入，确保新成员能快速访问到经过筛选的权威外部资料。
- **开源项目依赖参考页** 开源项目维护者可将项目所依赖的上游工具、镜像站、参考实现等外部链接汇总于此，方便用户查看，同时避免在 README 中堆砌过长 URL。
- **定期链接健康巡检** 借助内置的检查脚本，可定期对收录的站点进行可达性扫描，及时发现并清理失效域名，保持资源列表的有效性。

## 快速开始

```bash
# 克隆仓库
git clone https://github.com/your-org/qfys-navigator.git
cd qfys-navigator

# 安装依赖（仅用于检查脚本的标准库，无需额外安装）
# 若 Python 环境缺失 requests 等库，可手动安装，但非必须
python3 -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# 运行链接状态检查示例
python3 scripts/check_links.py --source resources/latest_batch.md

# 生成静态索引页（可选）
python3 scripts/build_index.py --input resources/ --output docs/index.md
```

## 贡献指南

1. 复刻本仓库到个人账号，并在本地克隆复刻后的版本，创建以 `batch-<编号>` 或 `fix-<描述>` 命名的特性分支。
2. 在 `resources/latest_batch.md` 中按现有格式追加或修改资源条目，确保每个 URL 严格保留原始字符串（不添加 `http://` 前缀，不修改域名大小写，不添加尾部斜杠）。若修改分类或标签，同步更新 `config/categories.yaml`。
3. 运行 `python3 scripts/check_links.py --source resources/latest_batch.md` 检查新添加的链接是否可达，并对超时或 4xx/5xx 状态的链接进行核实或标注。
4. 提交变更时使用清晰的 commit message，如 `feat: add batch 46 resources` 或 `fix: correct domain for qingniaov.xyz`，并推送到远程分支。
5. 通过 Pull Request 提交变更，在描述中说明本次新增或修改的资源数量、测试结果以及是否涉及文档同步更新。等待维护者审阅合并。

## 常见问题

**Q: 为什么资源列表中的 URL 没有统一加上 https:// 前缀？**  
A: 项目设计上严格遵循用户提供的原始格式，不自动补全协议或添加 `www.`。这样做的目的是保留用户原始意图，同时避免因猜测协议而导致跳转错误（部分站点仅支持 HTTP 或仅支持特定子域）。使用者复制链接后应自行判断协议。

**Q: 链接检查脚本报告某个站点超时，但浏览器中可以访问，怎么办？**  
A: 这可能是由于脚本默认超时时间较短（2 秒），或目标站点对非浏览器 User-Agent 有访问限制。您可以在 `scripts/check_links.py` 中调整 `timeout` 参数（如改为 5 秒），或添加 `--skip-timeout` 标志跳过超时检查。我们建议人工核实后仍保留该链接并添加注释说明。

**Q: 我是否可以删除历史批次归档中的链接？**  
A: 历史批次归档仅用于记录过往收录内容，一般不建议直接删除，以保持变更可追溯。如果某个链接确认为永久失效，可以在最新批次中将其移除，并在 `archive/` 对应批次文件中添加 `[DEPRECATED]` 标注，但保留原始条目供参考。
