# HyperLink Hub

HyperLink Hub 是一个面向技术内容创作者、开源文档维护者以及互联网信息聚合者的轻量级外链资源管理与导航系统。该项目并非传统的网址导航站，而是一个以 Markdown 为数据源、以版本控制为驱动、以静态站点生成技术为底层的结构化外链仓库。其核心定位是帮助技术团队或个人高效维护一批长期有效、分类清晰、可追溯变更的外部资源集合，尤其适用于 README 文档中的“相关链接”“友情推荐”“生态伙伴”等区块的自动化管理与展示。

目标用户包括开源项目维护者、技术博客作者、在线教育课程运营人员以及企业内部知识库管理员。这些用户在日常工作中经常需要引用大量外部 URL，但面临链接失效、分类混乱、更新时无法追溯历史、团队协作时难以同步修改等问题。HyperLink Hub 通过将外链资源以 YAML Front Matter 与 Markdown 列表相结合的方式存储，并配合自动化检查脚本与可视化看板，使得外链集合像代码一样可维护、可审查、可测试。

## 功能概览

**结构化外链数据存储** 所有链接以 Markdown 列表形式存储于 `/links` 目录下的分类文件中，每条链接支持附加标签、失效检测周期、备注说明等元数据，便于后续自动化处理。

**自动化链接活性检测** 内置基于 Node.js 的检测脚本，支持定期对仓库内所有外链发起 HEAD 请求，输出存活状态报告，并标记响应时间异常或状态码非 2xx 的链接。

**变更审计与版本追溯** 基于 Git 原生能力，每次对链接集合的新增、删除或修改均会留下提交记录与作者信息，支持快速回滚至任意历史版本，满足企业级审计需求。

**分类导航与多维度筛选** 提供按领域（如前端、后端、运维、AI）、按资源类型（文档、工具、社区、视频）、按维护状态（活跃、待确认、已失效）三种维度的筛选视图，所有视图均通过静态 HTML 生成，无需后端服务。

**团队协作工作流支持** 内置 GitHub Pull Request 模板与 Issue 表单配置，允许非维护者通过提交 PR 的方式推荐新链接，维护者可在合并前执行链接检测与内容审核。

**自定义输出模板引擎** 允许用户通过修改 `/templates` 目录下的 EJS 或 Handlebars 模板，自定义生成导航页面的 UI 风格，便于嵌入已有项目文档站点。

**命令行交互工具** 提供 `hlh-cli` 命令行工具，支持交互式添加链接、批量导入 CSV 格式链接表、导出失效链接清单等操作，提升日常维护效率。

## 应用场景

开源项目 README 中的“生态链接”区块维护。当一个开源项目发展到一定阶段，其周边工具、插件、适配案例会不断增多，维护者可以使用 HyperLink Hub 单独管理这些外链，并通过自动化构建流程将最新的链接列表渲染至 README 正文中，避免手动复制粘贴导致的错漏。

技术博客或教程网站的“参考资料”页面管理。技术教程往往需要引用大量官方文档、API 参考、社区讨论帖等外部资源，这些资源随时间可能发生 URL 重定向或内容迁移。使用 HyperLink Hub 的链接检测功能可以定期扫描所有参考资料链接，并在失效时发送邮件提醒博主更新。

企业内部知识库的“外部供应商”或“合作伙伴”信息看板。企业内部的运维或采购部门通常需要维护一份外部服务商列表（如云服务商、域名注册商、SSL 证书提供商等）。HyperLink Hub 支持私有化部署，且所有数据均为纯文本 Markdown，可轻松集成至企业现有的 GitLab 或 Gitea 服务中，实现权限管理与变更审批。

在线教育平台的“延伸阅读”模块。课程平台每节课末尾常附带若干延伸阅读链接，这些链接分散于不同章节，难以统一更新。HyperLink Hub 可以为每门课程建立一个独立的链接集合，并按章节打上标签，生成独立的延伸阅读页面，学生可通过该页面快速访问所有推荐资源。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，假设系统已安装 Git 与 Node.js 18.x 及以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/hyperlink-hub/hyperlink-hub.git
cd hyperlink-hub

# 安装项目依赖（包含核心检测引擎与 CLI 工具）
npm install

# 执行本地构建与链接检测，生成静态导航页面至 /dist 目录
npm run build
```

执行完成后，使用浏览器打开 `/dist/index.html` 即可查看默认样式的链接导航看板。若需自定义端口启动开发服务器，可执行 `npm run dev`，默认监听 `localhost:3000`。

首先，在 GitHub 仓库页面点击 Fork 按钮，将项目副本创建至您的个人账户下，随后克隆该副本至本地开发环境。请确保您的本地分支基于最新的 `main` 分支创建，建议使用 `feat/` 或 `fix/` 作为分支前缀以便识别。

其次，若您计划新增链接至 `/links` 目录下的某个分类文件，请严格按照该文件内已有的元数据格式进行添加，包括但不限于链接名称、URL、标签数组及添加原因备注。提交前请务必执行 `npm run check` 本地检测，确保新增链接处于可访问状态。

第三，若您的贡献涉及代码变更（如修改检测引擎、调整 CLI 逻辑或更新模板），请同时补充或更新对应的单元测试用例，并确保 `npm test` 全部通过。对于较大的功能改动，建议先在 Issues 中创建讨论主题，与维护者达成共识后再投入开发。

## 常见问题

**问：检测脚本是否会因为请求频率过高而被目标网站封禁？**

答：检测脚本默认采用串行请求模式，并发数可通过 `--concurrency` 参数调节，默认值为 3，即同时最多发起 3 个请求。每个请求均携带 `User-Agent: HyperLink-Hub-Checker/1.0` 头部，且遵循目标站点 robots.txt 中的 Crawl-delay 指示。对于大规模链接集合（超过 1000 条），建议将检测任务分散至多个时间片执行，或配置代理池以降低单 IP 请求密度。

**问：如何将 HyperLink Hub 集成至我已有的 VuePress 或 Docusaurus 站点中？**

答：HyperLink Hub 本身不限定于任何特定静态站点生成器。您可以通过两种方式集成：其一，在您的站点构建流程中增加一个步骤，调用 `hlh-cli export --format json` 将链接数据导出为 JSON 文件，然后由您的站点组件读取该 JSON 并渲染；其二，直接使用 HyperLink Hub 的 `templates/` 目录生成独立的 HTML 页面，并通过 iframe 或 `<object>` 标签嵌入您的站点中。详细的集成示例请参考 `/docs/developer-guide/integration.md`。

**问：链接失效检测的结果如何通知到维护者？**

答：检测结果默认输出至控制台，并同时生成一份 `reports/link-status.json` 文件。若配置了 GitHub Actions 的定时任务，检测完成后会自动将报告作为 Workflow Artifact 保存，并可选择通过 GitHub 的 Notification 系统或自定义的 Webhook 发送至企业微信、钉钉或 Slack 频道。邮件通知功能可通过配置 SMTP 环境变量启用，具体参数说明请见 `.env.example` 文件。
