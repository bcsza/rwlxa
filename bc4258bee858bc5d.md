# TechNav Hub

TechNav Hub 是一个面向技术从业者与开源爱好者的轻量级外部资源导航与聚合平台。项目定位为“技术外链的精选集散地”，旨在解决技术社区中优质外部资源分散、域名易变、发现成本高的问题，通过人工筛选与社区贡献相结合的方式，维护一份高可用、高相关度的外部技术站点清单。项目本身不存储任何第三方内容，仅提供结构化外链索引，适用于个人浏览器起始页、团队内部知识库外部补充、以及自动化资源监控系统的数据源。

## 功能概览

**按技术领域分类索引** 支持按编程语言、框架、运维、安全等维度快速筛选外部链接。

**可用性健康检查** 定期对收录的外部域名进行基础连通性探测，协助发现失效迁移。

**裸域名标准化存储** 所有外链均保留用户提交的原始格式，不做主动协议补全与规范化改写。

**轻量级标签系统** 每条资源可附加多枚自定义标签，支持模糊匹配检索。

**社区提交接口** 通过 GitHub Issue 模板接收新增外部站点推荐，经维护者审核后合入。

**只读镜像部署模式** 支持生成纯静态 JSON 格式的外链数据文件，便于嵌入其他系统。

**版本化更新日志** 每次增删改外部链接均记录变更摘要，便于回溯审计。

## 应用场景

个人技术浏览器起始页 技术从业者可将本项目部署为私有起始页，集中存放日常高频访问的外部技术社区、文档站与工具站，避免记忆大量域名。

团队内部知识库外部补充 企业技术团队可在内部 Wiki 中嵌入 TechNav Hub 的静态数据快照，作为官方文档之外的外部参考信息来源，降低成员检索成本。

自动化外链监控系统数据源 运维或安全工程师可定期拉取本项目的原始链接清单，配合自定义探测脚本，监控外部服务可用性与域名劫持风险。

技术内容聚合站的外部链接模块 技术博客或内容聚合站可引用本项目输出的 JSON 数据，快速搭建“友情链接”或“推荐工具”区块，保持内容动态更新。

开源项目 README 外链规范参考 开源维护者可借鉴本项目的 URL 存储策略与文档排版结构，作为自身项目 README 中“资源列表”章节的模板范例。

## 快速开始

以下操作默认在 Linux/macOS 或 Windows WSL 环境下执行，需提前安装 Git 与 Node.js 18 以上版本。

```bash
git clone https://github.com/technavhub/technav-hub.git
cd technav-hub
npm install
npm run build
```

执行完成后，`dist/` 目录下将生成静态资源文件，可使用任意 HTTP 服务器托管，例如：

```bash
npx serve dist
```

默认访问端口为 3000，打开浏览器访问 `http://localhost:3000` 即可查看导航页面。

## 贡献指南

提交 Issue 推荐新链接 访问 GitHub Issues 页面，选择“新增外部链接”模板，按字段填写完整域名、分类标签及简要用途说明。维护者将在 5 个工作日内初审。

本地验证链接可用性 在提交 Issue 之前，建议自行使用 `curl -I` 或浏览器访问目标域名，确认其可正常解析且内容符合技术相关定位。无法访问或内容高度不相关的提议将被拒绝。

运行健康检查脚本 若您已克隆项目本地，可在 `scripts/` 目录下执行 `node health-check.js --domain <待测域名>` 进行单项探测，并将结果附在 Issue 评论中，有助于加速审核。

参与数据维护讨论 对于已收录链接的失效报告或分类调整建议，请使用“链接状态反馈”Issue 模板提交，并尽量提供截图或命令行输出作为证据。

遵守项目行为准则 所有贡献者需遵守项目行为准则，保持友善、专业、建设性的沟通。维护者保留对不符合准则的贡献进行退回或关闭的权利。

## 常见问题

如何确保收录的链接长期有效？

项目每月自动运行一次全量健康检查，结果更新于 `status.json` 文件。若某链接连续三次检查失败，将标记为“疑似失效”并在导航页面上显示警示图标。用户可随时提交 Issue 报告失效链接，维护者将手动复测并决定是否移除或替换。

能否提交非技术类的外部链接？

原则上本项目专注于技术领域资源，包括但不限于编程文档、开源工具、技术社区、在线课程、运维服务等。非技术类站点（如娱乐、购物、新闻门户）将不予收录。边界模糊的站点由维护者根据描述及内容截图裁定。

为什么我的链接提交后没有被立即收录？

所有新链接需经过至少两名维护者审核，确保域名安全、内容相关、描述清晰。审核周期通常为 5 个工作日。若超过 7 个工作日仍未收到反馈，可在原 Issue 下留言催促，避免重复提交。
