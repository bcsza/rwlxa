# OpenResourceHub

OpenResourceHub 是一个面向技术调研人员、网络资源整理者与内容聚合站运维人员的开源外链资源导航项目。项目本身不托管任何盗版或侵权内容，仅对公开可访问的网络资源进行结构化整理与分类归档，提供稳定、可扩展的链接集合输出方案。目标用户包括：独立开发者、小型内容团队、网络数据分析师以及需要批量管理外部资源链接的技术运维人员。项目通过标准化章节模板、自动化链接校验脚本与多级分类索引，帮助用户从混乱的浏览器收藏夹中解脱出来，构建清晰、可维护、可共享的资源地图。

## 功能概览

- **多级分类索引**：支持按行业、地区、文件类型、更新频率等多维度对链接进行标记与筛选，输出层级清晰的索引目录，便于不同角色的团队成员快速定位所需资源。

- **链接健康度探测**：内置基于 HTTP 状态码与响应时长的周期性探测模块，自动标记失效链接、重定向链接及响应超时链接，并生成异常报告，降低人工巡检成本。

- **批量导入导出**：支持从 CSV、JSON 及标准浏览器书签 HTML 文件批量导入链接数据，同时支持导出为 Markdown 表格、结构化 JSON 或纯文本列表，适配不同发布平台。

- **变更日志追踪**：对资源库中每条链接的添加、删除、分类调整及 URL 变更均记录操作时间与操作人，生成完整的变更历史，方便回溯与审计。

- **自定义渲染模板**：提供多套 Markdown 渲染模板，用户可根据输出目标（GitHub README、静态站点、内部 Wiki）选择不同排版风格，并支持自定义 CSS 与页眉页脚内容。

- **权限分级管理**：内置基于角色的访问控制模型，支持管理员、编辑员、访客三级权限，编辑员仅可修改分类与备注，不可删除核心索引结构，管理员拥有完整操作权限。

- **定时同步任务**：支持配置 Cron 表达式，定时拉取指定源站或上游仓库的资源列表，自动合并新增链接并标记冲突项，减少人工同步工作量。

## 应用场景

- **技术调研期的外部资源汇总**：当团队需要对某一垂直领域（如流媒体技术、开源镜像站、学术数据库）进行系统性调研时，可使用 OpenResourceHub 快速搭建临时资源集合页，将零散发现的链接统一入库并添加调研备注，调研结束后一键导出完整报告。

- **小型内容站点的友情链接管理**：运营个人博客或小型行业资讯站的站长，可利用本项目的分类索引与健康度探测功能，定期检查友链存活情况，自动生成友链异常通知列表，避免站点出现死链影响用户体验。

- **多团队协作下的共享书签库**：在跨部门合作项目中，不同团队（如产品、开发、测试）需要共享一组公共测试环境链接、设计规范文档链接或运维监控面板链接。OpenResourceHub 的多级权限与变更日志功能确保链接库在多人协作下保持有序且可追溯。

- **离线文档的链接附录生成**：技术文档编写人员可将项目内部引用的所有外部链接统一托管至 OpenResourceHub，在撰写离线 PDF 或 CHM 文档时，从系统导出格式化链接附录，确保文档中引用的外部资源均有明确的来源与更新时间戳。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/your-org/OpenResourceHub.git

# 进入项目目录并安装依赖
cd OpenResourceHub
pip install -r requirements.txt

# 初始化本地配置文件并启动探测服务
cp .env.example .env
python main.py --init-db
python main.py --start-scheduler
```

## 贡献指南

1. 查阅 issue 列表或自行提出改进建议，在提交 Pull Request 前请在对应 issue 下留言说明认领，避免重复工作。对于新增分类或修改探测策略的贡献，需附带不少于 20 条真实链接的测试用例。

2. 克隆项目并创建以 `feature/` 或 `fix/` 为前缀的分支，所有代码变更需通过 `pytest` 全量测试套件，且新增功能需附带对应单元测试与集成测试。

3. 若涉及链接分类调整或新增资源类别，请同步更新 `config/taxonomy.yaml` 文件，并在 `docs/taxonomy_design.md` 中补充分类依据与使用建议。

4. 提交 Pull Request 时，请按模板填写变更摘要、测试覆盖情况以及是否影响现有用户配置。合并前需至少一位项目维护者进行代码审查和功能验证。

5. 文档类贡献（包括错别字修正、示例补充、翻译）无需单元测试，但需确保 Markdown 语法正确且与当前版本功能一致，可直接提交 Pull Request 并标注 `[doc]` 前缀。

## 常见问题

Q: 探测引擎运行时报错 `SSL: CERTIFICATE_VERIFY_FAILED`，如何处理？

A: 部分外部资源站点使用自签名证书或过期证书。您可在 `probe_policy.yaml` 中将 `verify_ssl` 设置为 `false` 以跳过证书验证，但请注意这会降低安全性。更推荐的方式是使用 `ssl_context` 参数指定自定义 CA 证书包路径。

Q: 数据库中的链接状态长时间未更新，如何手动触发全量重探测？

A: 执行 `python main.py --probe --force` 命令，该参数将忽略调度缓存，对所有状态为 `active` 和 `unknown` 的链接发起实时探测。若仅需探测特定分类，可追加 `--category` 参数。

Q: 能否将项目部署为多用户 Web 服务而非仅 CLI 工具？

A: 项目核心库与数据库模型已支持多用户权限，但 Web 管理界面（基于 Flask）处于可选组件状态。如需启用，请安装 `flask` 与 `flask-login` 依赖，并参考 `docs/operations.md` 中关于 `web_mode` 的配置章节，启动 `python web_app.py` 即可运行内置管理面板。
