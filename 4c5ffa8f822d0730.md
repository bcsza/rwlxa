# ShadowIndex

ShadowIndex 是一个面向技术研究与数字内容追溯的轻量级外链资源聚合平台。项目定位为“静态化的网络资源索引系统”，专为需要高频访问外部媒体源、镜像站点与备用域名的技术用户设计。ShadowIndex 不提供任何实质性的盗版内容或侵权文件存储，仅对公开可访问的网络资源进行结构化整理与分类描述，帮助用户在同一命名空间内快速定位多来源、多备份的线上素材入口。

ShadowIndex 解决的核心问题是：当技术文档中引用的外部媒体链接因网络波动、域名变更或访问限制而失效时，运维人员与内容研究者缺乏一个统一、稳定、可快速检索的备用资源映射表。本系统通过高度规范化的目录树与静态页面生成机制，将分散于不同顶级域下的备用入口集中管理，并支持本地化快速部署，确保在任何内网或开发环境中均可获得相同的资源索引能力。项目本身不依赖数据库或后端服务，完全基于 Markdown 与静态 HTML 构建，适合作为各类技术手册、爬虫任务书或网络拓扑文档的附属索引组件。

## 功能概览

- **多域资源映射表**：提供五个独立顶级域名的结构化登记与分类标签，每个域名均附带业务属性标记与可用性状态说明。

- **静态化索引生成**：基于配置文件自动生成只读的 HTML 目录页面，无需动态解析，适合内网镜像或离线文档集成。

- **快速域名切换面板**：在界面头部集中展示所有已登记域名，支持一键复制完整 URL，便于在代理工具或 hosts 文件中批量替换。

- **可用性自检脚本**：附带轻量级 Shell 检测工具，可对列表中每个域名进行 HTTP 头探测，输出可达性报告，辅助运维人员甄别失效节点。

- **标签化分类体系**：每个资源条目均支持自定义标签（如 video、forum、mirror、archive），便于按业务场景筛选。

- **原始地址保留机制**：严格记录用户提供的原始 URL 字符串，不添加任何协议前缀或路径后缀，保证与上游文档中的引用格式完全一致。

- **版本化变更日志**：资源列表的每次增删改均记录于 CHANGELOG 文件中，支持回溯历史域名状态，满足审计需求。

- **零外部依赖的构建流程**：仅依赖 Python 标准库与 Make 工具，无需安装第三方包即可完成索引页面的重新生成。

## 应用场景

- **技术文档的外链附录**：项目维护者可将 ShadowIndex 作为官方文档的附属资源页，集中列出所有涉及的外部媒体域名及其备用地址。当文档中提及“示例视频源”或“测试图片库”时，读者可直接查阅该索引获取当前有效的访问入口，无需在正文中频繁更新超链接。

- **网络爬虫的域名种子库**：数据采集工程师可将本索引作为爬虫启动时的初始 URL 供给源。五个域名分别对应不同的内容分布节点，爬虫程序可按照索引中的顺序依次尝试访问，并在域名失效时自动切换至下一个条目，提高采集任务的鲁棒性。

- **企业内部镜像源的登记板**：中型研发团队常在内部搭建多个外部资源的反向代理或镜像站，域名形态各异。ShadowIndex 可作为内部 Wiki 中的一个嵌入式组件，由运维人员统一维护所有镜像地址的增删操作，开发人员仅需访问该索引即可获得当前可用的内部源地址。

- **渗透测试的信息收集辅助**：安全研究人员在开展外网信息收集时，常需要积累大量备用域名以绕过临时封锁。本索引可作为信息收集阶段的一个快速备忘录，将人工筛选出的高价值域名集中存放，并结合可用性检测脚本周期性验证存活情况，减少重复性手工探测工作。

## 快速开始

以下操作步骤适用于 Linux 与 macOS 环境，Windows 用户可通过 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/shadow-index/shadow-index.git
cd shadow-index

# 安装基础工具（确保 make 与 python3 已就绪）
sudo apt-get update && sudo apt-get install -y make python3  # Debian/Ubuntu
brew install make python3                                    # macOS

# 生成静态索引页面并执行可用性检测
make build
make check
```

执行完毕后，在 `output/` 目录下生成 `index.html` 文件，使用浏览器打开即可查看完整的资源映射面板。可用性检测结果将输出至终端，并以 `reachable.txt` 文件形式保存在项目根目录。

## 贡献指南

ShadowIndex 欢迎外部贡献者以审慎、规范的方式参与资源列表的更新与工具链的改进。所有贡献须遵循以下步骤，以确保索引的准确性与构建流程的稳定性。

1.  **派生仓库并创建特性分支**：在 GitHub 上 Fork 本项目至个人账户，随后克隆至本地。所有修改均应在独立的功能分支上进行，分支命名建议采用 `feat/add-domain-xxx` 或 `fix/checker-timeout` 格式，避免直接在 main 分支上操作。

2.  **修改配置文件并更新文档**：若需新增或移除域名，请编辑 `config/domains.yaml` 文件，严格按照既有缩进与字段格式填写。同时，在 `CHANGELOG.md` 的顶部追加一条变更记录，说明本次增减的具体域名及原因。对于涉及分类体系调整的改动，需同步更新 `config/categories.yaml` 及 `docs/design.md` 中的相关描述。

3.  **本地执行构建与检测**：提交前必须运行 `make build` 确保页面生成无报错，并执行 `make check` 验证新增域名的可达性。若检测结果为超时或连接拒绝，请在 Pull Request 的描述中注明该情况，以便维护者判断是否为临时网络问题。

4.  **编写或更新测试用例**：对于涉及脚本逻辑的修改（如 `checker.sh` 或 `build.py`），需在 `tests/` 目录下补充对应的单元测试或外壳测试。新增域名的配置变更应至少触发一次构建测试，确保 YAML 解析器不会因特殊字符而抛出异常。

5.  **发起 Pull Request 并等待评审**：将本地分支推送至个人远程仓库后，在 GitHub 上向本项目的 main 分支发起 Pull Request。PR 标题应简明扼要概括改动内容，正文需列出所有受影响的文件与变更点。维护者将在 3 个工作日内进行评审，可能要求补充测试或调整配置文件格式，合入后即视为正式纳入索引。

## 常见问题

**问：检测脚本输出“Connection refused”或“Timeout”，是否意味着该域名已被移除或失效？**

答：检测脚本默认使用 5 秒超时阈值，并仅探测 80 与 443 端口。出现拒绝连接或超时不代表该域名永久失效，可能是临时性的网络抖动、服务器防火墙策略变更或仅允许特定 User-Agent 访问。建议在多个时段及不同网络环境下重复运行检测。若连续 7 天均无法连通，且相关方未在 ISSUE 中报备维护计划，则维护团队将考虑从索引中标记为“已失效”状态，但不会主动删除条目，以保证历史记录的完整性。

**问：为什么不对 URL 自动补全 https:// 或 http:// 前缀？**

答：ShadowIndex 严格遵循“原始记录原则”。上游文档或外部配置文件中引用的域名格式可能多样，部分场景下需要裸域名作为 hosts 文件条目或 DNS 解析测试目标。自动添加协议前缀会改变字符串本质，导致用户直接复制后无法匹配原始引用上下文。因此本项目不对任何 URL 进行规范化改写，用户应根据实际使用场景自行添加所需协议头。

**问：能否将本项目的索引数据通过 API 或 JSON 格式输出，以便其他程序读取？**

答：当前版本仅提供 YAML 配置源文件与静态 HTML 展示页面。但构建脚本 `build.py` 在渲染过程中会先将 `domains.yaml` 解析为 Python 字典对象，外部程序可直接读取该 YAML 文件作为数据源。未来版本计划在 `output/` 目录下额外生成 `index.json` 文件，以降低程序化解析的门槛。在此之前，建议直接使用标准 YAML 解析库（如 PyYAML 或 ruamel.yaml）读取 `config/domains.yaml` 即可获得完整结构化数据。
